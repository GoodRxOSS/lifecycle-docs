The `environment` section controls how Lifecycle deploys and manages your ephemeral environments. It defines which services are deployed by default, which are optional, and how automation (webhooks) should behave.

## Fields Reference

| Field                    | Type    | Required | Description                                             |
| ------------------------ | ------- | -------- | ------------------------------------------------------- |
| `autoDeploy`             | boolean | No       | Automatically deploy on PR updates                      |
| `defaultServices`        | array   | Yes      | Services deployed with every environment                |
| `optionalServices`       | array   | No       | Services available on-demand                            |
| `agentSession`           | object  | No       | Agent session overrides for this repository/environment |
| `webhooks`               | array   | No       | Automation triggers on state changes                    |
| `enabledFeatures`        | array   | No       | Feature flags for the environment                       |
| `githubDeployments`      | boolean | No       | Create GitHub deployment records                        |
| `useGithubStatusComment` | boolean | No       | Post status comments on PRs                             |

## defaultServices

Services listed in `defaultServices` are deployed automatically when an environment is created. At least one default service is required.

```yaml filename="lifecycle.yaml"
environment:
  defaultServices:
    - name: "api" # Required - service name (must match a service definition)
    - name: "frontend"
    - name: "database"
```

### Service Reference Fields

Each entry in `defaultServices` can include optional fields to override the service configuration:

| Field        | Type   | Description                                                   |
| ------------ | ------ | ------------------------------------------------------------- |
| `name`       | string | **Required.** Name matching a service in the `services` array |
| `repository` | string | Override the repository for this service                      |
| `branch`     | string | Override the branch for this service                          |
| `serviceId`  | number | Internal service ID reference                                 |

```yaml filename="lifecycle.yaml"
environment:
  defaultServices:
    - name: "api"
      repository: "myorg/api" # Optional - override repository
      branch: "develop" # Optional - override branch
```

These overrides also affect Agent Sessions and sandbox sessions:

- Sandbox service discovery starts from `defaultServices` and `optionalServices`.
- `repository` and `branch` tell Lifecycle which repo config to read when a
  service is selected for dev mode or sandboxing.
- Keep selectable service names unique within an environment so the UI can tell
  them apart.

## optionalServices

Services listed in `optionalServices` are not deployed by default but can be added to an environment on-demand through the Lifecycle UI or API.

```yaml filename="lifecycle.yaml"
environment:
  defaultServices:
    - name: "api"
  optionalServices:
    - name: "cache"
      repository: "myorg/services"
      branch: "main"
    - name: "worker"
      repository: "myorg/services"
      branch: "main"
```

<Callout type="info">
  Optional services are useful for components that aren't always needed, like
  background workers, caching layers, or additional databases for specific
  testing scenarios.
</Callout>

## autoDeploy

When `autoDeploy` is enabled, Lifecycle automatically triggers a deployment whenever the PR is updated (new commits pushed).

```yaml filename="lifecycle.yaml"
environment:
  autoDeploy: true # Default: false
  defaultServices:
    - name: "api"
```

<Callout type="info">
  With `autoDeploy: true`, environments are created automatically on every
  commit to the PR branch without requiring manual label addition.
</Callout>

## webhooks

Webhooks allow you to trigger external actions when the environment reaches certain states. See the [Webhooks](/docs/schema/webhooks) page for detailed configuration.

```yaml filename="lifecycle.yaml"
environment:
  defaultServices:
    - name: "api"
  webhooks:
    - name: "Run E2E Tests"
      state: deployed
      type: docker
      docker:
        image: "myorg/e2e-tests:latest"
      env:
        TEST_URL: "https://{{api_publicUrl}}"
```

## agentSession

Use `environment.agentSession` for repository-level agent session overrides that
should apply to both environment-backed sessions and sandbox sessions launched
from this environment configuration.

Today, the supported override is `resources`, which lets you tune the default
CPU and memory requests and limits for the agent and editor containers.

```yaml filename="lifecycle.yaml"
environment:
  defaultServices:
    - name: "api"
  agentSession:
    resources:
      agent:
        requests:
          cpu: "1"
          memory: "2Gi"
        limits:
          cpu: "2"
          memory: "6Gi"
      editor:
        requests:
          cpu: "250m"
          memory: "512Mi"
        limits:
          cpu: "1"
          memory: "1Gi"
```

Behavior:

- These values override the platform defaults from
  `global_config.agentSessionDefaults.resources`.
- They apply to the agent session runtime, not to your normal deployed
  application workloads.
- Service-specific readiness overrides belong under
  `services[].dev.agentSession.readiness`, not under `environment.agentSession`.

## Agent Session and sandbox service resolution

Use explicit service references when your environment spans multiple
repositories or branches.

- `defaultServices` and `optionalServices` define the pool of services that
  sandbox sessions can fork from the base environment.
- Add `repository` and `branch` when a service comes from a different repo than
  the environment source, or when the same service name appears more than once
  across active deploys.
- Keep selectable service names unique within one environment so the UI can
  distinguish them cleanly.
- If a selected service's `requires` dependencies cross repo or branch
  boundaries, add `repository` and `branch` to those dependency references too
  so sandbox dependency resolution stays unambiguous.

Example:

```yaml filename="lifecycle.yaml"
environment:
  defaultServices:
    - name: "web"
      repository: "example-org/ui"
      branch: "feature/ui"
    - name: "api"
      repository: "example-org/platform"
      branch: "feature/api"
  optionalServices:
    - name: "worker"
      repository: "example-org/platform"
      branch: "feature/api"
```

```yaml filename="lifecycle.yaml"
environment:
  autoDeploy: true
  githubDeployments: true
  useGithubStatusComment: true

    - name: "frontend"
    - name: "database"

  optionalServices:
    - name: "cache"
      repository: "myorg/infrastructure"

    - name: "Integration Tests"

      type: docker

    - name: "Slack Alert"
      state: error
      type: command
      command:
        image: "alpine:latest"
        script: |
          apk add --no-cache curl
          curl -X POST "$SLACK_WEBHOOK" \
            -H "Content-Type: application/json" \
            -d '{"text":"Deployment failed for PR"}'
      env:
        SLACK_WEBHOOK: "https://hooks.slack.com/services/XXX/YYY/ZZZ"
```
